HelixScript Field Guide

HelixScript is the beginner-friendly language for describing what seeds HelixSeed should search. The app compiles this script into the scanner query internally.

← Back to the User Guide

1. Start With The Basic Shape

  1. Use anchor to choose the main point, usually origin.
  2. Use logic and when every rule must match.
  3. Use find to add structure rules.
  4. Press Run in HelixSeed. The app compiles HelixScript before launching the native scanner.

2. Minimal Scan

anchor origin
logic and

find village as v1 within 64 of origin strict

This searches for a village within 64 blocks of origin. v1 is the rule id and can be reused by later commands.

3. Variables

let structure = village
let radius = 64

find $structure as v1 within $radius of origin strict

Variables are inserted with $name. Use them for values you expect to tweak often, like radius, structure names, or mode choices.

Numeric assignments accept simple math: + - * / %, parentheses, and the helpers min, max, abs, floor, ceil, round.

let base = 64
let wide = base * 2
let snug = max(32, base - 16)

Use const name = value when you want the assignment to stay locked. set is an alias for let.

4. Lists

let biomes = [plains, meadow, cherry_grove]
let radii = [64, 96, 128]

biome origin y 80 radius 128 allow $biomes

Lists are written with square brackets. Items can be names, numbers, or quoted strings. $biomes by itself joins the items with commas, which is exactly what biome filters and other comma lists expect.

let biomes = [plains, meadow]

# .length, .first, .last and [index] read individual pieces
find village as v1 within ${radii[0]} of origin strict
assert $biomes.length > 0 message need at least one biome

Use list contains item in conditions for membership checks. The same operator works on plain strings as a substring test.

if $biomes contains plains
  biome origin y 64 radius 64 allow plains
end

5. For Loops

for radius in [64, 96, 128]
  find village as v_$radius within $radius of origin strict
end

A for <name> in <list> block repeats its body once per item, binding the item to $name. The list can be an inline literal or a previously declared variable.

let biomes = [plains, meadow, cherry_grove]

for biome in biomes
  biome origin y 80 radius 128 allow $biome
end

Loops unroll at compile time, so each iteration produces independent commands — handy for generating one rule per structure or one biome filter per area.

6. Conditions

let radius = 96
let structure = village

if radius >= 64 and structure is village
  find $structure as v1 within $radius of origin strict
else
  find village as v1 within 32 of origin placement
end

Conditions support and, or, not, is, is not, contains, ==, !=, >, >=, <, and <=.

if radius >= 96
  find village as v1 within $radius of origin strict
elif radius >= 48
  find village as v1 within $radius of origin placement
else
  stop
end

Chain branches with elif <cond> (or else if <cond>). Only the first matching branch runs. else must come last.

Inside an if/elif/else/skip block, use continue (optionally continue if <cond>) to early-exit just the current branch, or pause (optionally pause if <cond>) to bail out of every enclosing block back to the top level. stop still ends compilation entirely.

7. Rule Field Lookups

find village as v1 within 45 of origin strict

if $radius of v1 == 45 and $structure of v1 is village
  find ruined_portal as rp1 within 96 of constraint:v1 strict
end

Use $field of id when a condition needs to read a specific rule. Supported fields are radius, structure, mode, anchor, and id. The referenced rule must appear before the if.

find village as v1 within 64 of origin placement optional
if v1 == found
  find ruined_portal as v2 within 64 of constraint:v1 placement
  skip
end

found is a runtime per-seed check: it asks whether the scanner actually located that structure in the current seed. Mark a find as optional so its absence does not reject the seed by itself — without optional, every seed must satisfy it, and the check is always trivially true.

Inside an if <id> == found block (also !=, joined by and / or), only three body forms are allowed and each lowers to a single fixed-form runtime opcode:

  • skip — reject this seed (no seed-level allocation, just a flag check)
  • stop — halt the entire scan, same effect as pressing Stop in the UI
  • a single find line — conditionally require that structure only when the predicate is true

Up to 31 rule ids can be referenced by found in one query. elif, else, nested if, or other statements inside an if <id> == found block are rejected at compile time.

8. Structure Rules

find village as v1 within 64 of origin strict
find ruined_portal as rp1 within 96 of constraint:v1 strict
find trial_chambers as tc1 within 256 of spawn placement

strict asks for exact validation. placement is faster but less exact. Anchors can be origin, spawn, or constraint:id.

9. Biome Filters

biome origin y 64 radius 128 allow plains,meadow,cherry_grove

Biome filters check an area around a point. Use comma-separated or space-separated biome names after allow.

10. Output

output
output raw
output both
output lift64
output world64

Use output by itself to print found seeds in the normal raw seed mode. Use a mode after it when you need lifted 64-bit output or both raw and lifted values.

11. Functions

function nearby_portal(anchor_id, portal_id, radius)
  find ruined_portal as $portal_id within $radius of constraint:$anchor_id strict
end

find village as v1 within 64 of origin strict
nearby_portal(v1, rp1, 96)

Functions are compile-time helpers. They expand before HelixSeed launches the scanner, so they are good for reusable query patterns without adding runtime overhead.

12. Scan Control

scan
  start 0
  count 1000000
  batch balanced
  max_print 20
  progress 1
end

assert $radius of v1 > 0 message v1 radius must be positive
skip if $mode of v1 is placement
  find ruined_portal as rp1 within 96 of constraint:v1 strict
end
stop if $radius of v1 > 512

scan sets runtime options from the script. assert fails fast, skip if skips a block when true, and stop if stops compiling the rest of the script.

13. Performance Block

performance
  workers 0
  execution auto
  stage auto
  cap adaptive
  surrogate balanced
  fixed_cap 128
end

Keep these values on auto unless you are profiling. Use the Telemetry tab to compare changes.

14. Layout & Indentation

HelixScript is line-based: every command is a single line, and whitespace at the start of a line is ignored. Indenting block bodies is encouraged for readability but never required.

# both of these compile to the same thing
if radius >= 64
  find village as v1 within $radius of origin strict
end

if radius >= 64
find village as v1 within $radius of origin strict
end

Use # or // to start a comment, and tab or two-space indents inside blocks if you find them easier to read.

15. Block Menu

The block menu beside the editor inserts common script pieces: Variable, List, For, If, Assert, Skip, Stop, Output, Function, Scan, Find, Biome, and Performance. Click a block, then edit the placeholders.

16. Common Mistakes

  • Define variables before using $variable.
  • Close every if, for, skip, scan, function, and performance block with end.
  • Use rule ids like v1 with constraint:v1 anchors.
  • Use lowercase names with underscores for structures and biomes.
  • For loop variables can be reused (for r in radii), but they must be plain let names — not const ones.
